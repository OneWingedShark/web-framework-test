Overview and main concepts

Mneson is an Ada library for persistent graphs of basic values (integer, string, float). A vertex may also be valueless.

A graph is a set of links. A link is an ordered pair (source, target) of vertices. Links are untyped. Graphs can be cyclic.

A vertex is a small (80 bits) structure capable of representing a basic data value. During processing (graph open) all linked vertices are in main memory. Upon closing or saving all data is stored in files. Values whose size exceeds the vertex structure size are partially stored in files at all times. A vertex may also be valueless i.e contain no value at all.

The library is construed as a generic package called Mneson. An instance of Mneson represents a graph.

A Mneson graph is a pure ADT (Abstract Data Type) i.e. its state is manipulated only through operations.

Mneson calculus

The type Selection represents a set of vertices in the Mneson calculus. A selection is an intensional representation. Operation For_Each materializes each element of the selection iteratively. The Mneson Calculus primitives are Targets, Sources, Intersect.

File operations

A Mneson graph is either closed or open. A closed graph is in an inconsistent state regarding its content.

Because files are involved, open operations require a name that must be a valid filename. Standard exceptions e.g. Name_Error associated with standard file operations may be propagated upon passing an invalid name.

Create attempts to open a new empty graph under the given name. If the current graph is open then State_Error is propagated, and the graph is not changed.

Open attempts to open the named graph. If the current graph is open then State_Error is propagated, and the graph is not changed.

Save closes the graph and then reopens it with the same name. Upon success, the graph is not changed.

Save_As closes the graph, copies the associated physical files to the new name, and then opens this newly named graph. Upon success, the graph is not changed, except for its name.

Close attempts to close the current graph. If the current graph is already closed then State_Error is propagated, and the graph is not changed (but it is in an inconsistent state).

Vertex operations

Function New_Vertex returns a new unique valueless vertex.

Each function To_Vertex returns a unique vertex representing the given value.

A vertex holding a value of type T is called "a T vertex".

Predicates Is_Integer, Is_String, etc. can be used to inspect the type of value held in a vertex.

Function Img returns an image of the given vertex. The image of an integer vertex is equal to the standard Integer'Image result. The image of a string vertex is the contained string value enclosed in double quotes (i.e. including the quotes). The image of a valueless vertex is the character '#' followed by the decimal representation of the vertex number which is a positive number that uniquely identifies the vertex within its graph; this representation, unlike the standard Positive'Image, does not have a leading space character. Examples: 

 123
-456
"abc"
#789

The type predicates above and Img are considered 'utility' functions. They are not designed for graph management. See Mneson/T for properly typed graphs

Link operations

Links are the heart and soul of Mneson. Only links are created or destroyed. Only linked vertices are stored. Only linked vertices can be retrieved. Link operations are organized in families: connect, disconnect, reconnect, for-each, get-vertices.

Planned increments

Mneson/P

P for "Protected", or "Paralell". A 'thread-safe' version of Mneson. I think this can be accomplished with a protected object containing all operations renamed to the 'unprotected' ones. This protected object can be nested in or a children of package Mneson (Mneson.P) or a separate package withing Mneson (Mneson_P).

Mneson/T

T for "Typed graphs". This can be accomplished by construing typed links as vertices:

procedure Connect (Source, Link, Target : Vertex);

It seems wise to base Mneson/T on Mneson/P.

Files in the distribution

Source code files follow the GNAT convention: one library unit per file, lowercase filenames equal to the unit name with hyphens instead of dots, "ads" extension for specs, "adb" extension for bodies.

Package Mneson constitutes the library. Package Order and certain descendants of AI302 are required by Mneson. All are included in the distribution.

Demonstration

Let's use the Test.Interactive tool to create and explore a graph representing the following toy database of contacts:

NameEmail addresses
marius amado.alves@netcabo.pt
maa@liacc.up.pt
liacc maa@liacc.up.pt
comp.lang.ada comp.lang.ada@ada-france.pt

First invoke the tool (the details here depend on the operating system; the example shown is on Linux; on Windows, type test-interactive.exe or click on the icon): 

$ ./test-interactive
This program provides interactive Mneson graph testing.
The prompt shows the current graph name and state.
Type HELP for the list of available operations.
Mneson homepage: www.liacc.up.pt/~maa/mneson
TEST CLOSED>

Note the graph is closed, hence inconsistent. Let's create a new graph called "demo" for our demonstration: 

TEST CLOSED>create demo
demo OPEN>

Now we can populate the graph (the LINK command equates the Connect operation): 

demo OPEN>link "marius" "amado.alves@netcabo.pt"
demo OPEN>link "marius" "maa@liacc.up.pt"
demo OPEN>link "liacc" "maa@liacc.up.pt"
demo OPEN>link "comp.lang.ada" "comp.lang.ada@ada-france.org"

By now we can already retrieve email addresses given a name: 

demo OPEN>show "marius"
"marius"
|__"amado.alves@netcabo.pt"
|__"maa@liacc.up.pt"

To show all contacts we have to link them all to some vertex. Let's call it "top": 

demo OPEN>link "top" "marius"
demo OPEN>link "top" "comp.lang.ada"
demo OPEN>link "top" "liacc"

Let's verify: 

demo OPEN>show "top"
"top"
|__"comp.lang.ada"
|__"liacc"
|__"marius"

We can also show all names along with their email addresses. Just specify a maximum path length greater that the default 1: 

demo OPEN>show "top" 2
"top"
|__"comp.lang.ada"
|  |__"comp.lang.ada@ada-france.org"
|__"liacc"
|  |__"maa@liacc.up.pt"
|__"marius"
   |__"amado.alves@netcabo.pt"
   |__"maa@liacc.up.pt"

Let's call it a day: 

demo OPEN>quit
$

Note QUIT properly closes the graph before terminating. Confirm this by reinvoking the tool and issuing "open demo" (without the quotes).

Mneson Conventions

The Mneson Conventions define a way to represent complex structures, notably attributes and typed links, in the untyped graph.

Basic attribute representation convention

1. For each attribute type, there exists a vertex that represents it.

2. Each target of an attribute type is an attribute instance.

3. An attribute instance has exactly 2 sources.

4. Of necessity, one is the attribute type. The other is the subject of the attribute.

5. The targets of an attribute instance represent its value.

Alternate valueless attribute instance representation convention

6. An atribute type represents a valueless attribute instance.

Simple attribute type naming convention

7. The target of a string s is an attribute type of name s.

Simple typed connection representation convention

8. An attribute instance represents an typed connection.

Simple set representation convention

9. For a vertex that represents a set, its targets represent its elements.

Simple record representation convention

10. A record is a set of attribute instances, the components of the record.

Simple vector representation convention

11. A vector is a one dimensional array of integer index. 12. A vector of index range [First, Last] is represented as a record whose components are of type First to Last.

Why attribute instances

And not attribute values directly represented in their place.

Because if they were, then the attribute types of distinct subjects with the same value could not be distinguished.

Mntext

Mntext is a textual language for the description of graph data. A Mntext item describes an attribute instance following the Mneson conventions.

Mntext grammar

item ::= [basic_value_sequence] value
basic_value ::= name / number / text
value ::= basic_value / set
set ::= '{' item_list '}'

Semantics

A Mntext compiler accepts an open graph and a Mntext item and returns a (possibly created anew) vertex that represents the described attribute instance. The choice of a subject for this attribute instance is outside the scope of Mntext.

Examples

{marius {amado.alves@netcabo.pt, maa@liacc.up.pt},
 liacc {maa@liacc.up.pt},
 comp.lang.ada {comp.lang.ada@ada-france.org}}
Or, with a more precise schema: 
top{

  {name liacc,
   email {maa@liacc.up.pt}},

   email {comp.lang.ada@ada-france.org}}}
